Techniques for making tables more accessible

One major component of the WCAG guidelines is providing table headers for data tables, and associating cells of data within those tables with the headers that they relate to. Especially when navigating unsighted, data tables can be daunting as without a full view of the table - and therefore without context - it can be difficult to ascertain just what a data cell's meaning is.

What is a data table?

HTML_CodeSniffer currently presumes that all tables are data tables. This is based on the presumption that new modern site designs are likely to use a layout design based upon CSS. However, certain tests will throw a warning that will apply only if a table is a data table.

The current HTML5 working draft contains a suggested method of determining whether a table is a data or layout table:

User agents, especially those that do table analysis on arbitrary content, are encouraged to find heuristics to determine which tables actually contain data and which are merely being used for layout. This specification does not define a precise heuristic, but the following are suggested as possible indicators:

FeatureIndication
The use of the role attribute with the value presentationProbably a layout table
The use of the border attribute with the non-conforming value 0Probably a layout table
The use of the non-conforming cellspacing and cellpadding attributes with the value 0Probably a layout table
The use of caption, thead, or th elementsProbably a non-layout table
The use of the headers and scope attributesProbably a non-layout table
The use of the border attribute with a value other than 0Probably a non-layout table
Explicit visible borders set using CSSProbably a non-layout table
The use of the summary attributeNot a good indicator (both layout and non-layout tables have historically been given this attribute)

HTML5: The table element - working draft, hence work in progress

Use of scope and/or headers attributes

There are two techniques provided for associating headers with data cells:

  • the scope attribute, which associates a table header (th element) with a single row or column (or, in theory, a group of them); and
  • the headers attribute, which associates a data cell (td element) with one or more table header cells (identified by ID, which must be assigned to each header cell).

HTML_CodeSniffer recognises both methods, and will warn users if either of them should be used for a data table. Broadly speaking, this will occur when a table contains both row and column headers, or when a single set of headers spans multiple levels.

Required attributes to disambiguate table headers' meaning
Type of tableRequirementsReason
Two or more levels of either row and/or column headersMust use headers attribute The scope attribute is ambiguous with multiple levels of column headers, and the "rowgroup" and "colgroup" values of the scope attribute appear to not be accepted enough (in screen readers, for example) to be considered "accessibility-supported".
Single-level row header and single-level column headerEither method is required The "row" and "col" values of the scope attribute are sufficient to unambiguously associate a data cell with headers, and are accessibility-supported. Alternatively, the headers attribute can still be used.
Single-level row header or single-level column headerUse of either method optional The header meanings are unambiguous in such a table, so such methods are not required.
Detecting incorrect headers attributes

When the headers attribute technique is used and th elements are properly identified with IDs, HTML_CodeSniffer will calculate the expected value of the headers attribute for each cell, and emit an error for each cell that fails this comparison. The headers attribute is specified as a space-separated list of IDs, similar to the class attribute: order is not important, and HTML_CodeSniffer will accept cells where the headers attribute is specified correctly but in a different order to that expected.

HTML_CodeSniffer will not throw errors on td elements that appear before the first row and column of headers. For instance, if a table contains a cell in row 1, column 1, with no meaning and thus given a td element instead of th, no error will be thrown. (This example is written independent of direction - normally this will be the top-left table cell, but in a right-to-left (RTL) language it will be the top-right.)

Adding summary and caption to tables

Adding a caption and summary provide a way to provide further information on identifying and using the table. The two relevant techniques are:

  • H39: Using caption elements to associate data table captions with data tables, and
  • H73: Using the summary attribute of the table element to give an overview of data tables.

The caption element is mainly to programmatically associate a table with its caption. The method technique H39 uses suggests that a table does not need a caption to fulfil this Success Criterion, but if they exist, they should be associated using the caption element. It also ensures that the caption element is not used for other purposes. HTML_CodeSniffer will emit a warning if a caption is missing.

The summary attribute is more important, as it provides information on the use of a table, especially to non-sighted users where use of a table is not obvious. The W3C article provides the example of a bus timetable, which uses the summary attribute to explain how to read it (in this case: bus stops are listed in row 1; find your bus stop then look down that column to find departure times). For sighted users reading the timetable would be obvious, but someone with a screen reader who cannot skim-read the table first would not have this orientation.

The summary attribute's main issue is that it is not shown to sighted users, so if the information that would appear in the summary attribute would also be useful to other groups of users, it may be better placed in body text before the table (not after, because it is not obvious to those users reading sequentially). Therefore, the lack of a summary is reported by HTML_CodeSniffer as a warning.

The two techniques also state that where both caption and summary are specified, the information should not be duplicated.

Making forms more accessible

Labelling form elements

Labels must be explicitly labelled, using the "for" attribute, according to H44: Using label elements to associate text labels with form controls. If the visual design cannot accommodate a label, then you can use the title attribute instead - see H65: Using the title attribute to identify form controls when the label element cannot be used.

Implicit labels are where the label attribute wraps around the field: they are not supported by assistive technology as well as explicit labels, so are not recommended. The input field becomes part of the label, which makes it more difficult to determine what the label actually is; it is not as accessibility-supported in older versions of assistive technology.

Using the label element is preferred whenever possible, because sighted users relying upon the keyboard for navigation may not be able to access the title attribute, which normally appears as a tooltip in the browser.

Ensuring label targets exist in the document

HTML_CodeSniffer will emit an error message on a label element in the following circumstances:

  • The "for" attribute contains an ID that does not exist in the current document; or
  • The "for" attribute contains an ID that points to a form control.

For the latter test, specifically tested are input, select and textarea elements. Button elements are not tested, as these should not have a label (their caption acts as its label).

Grouping form elements using the fieldset element

Fields that are logically related should be grouped using the fieldset element, according to technique H71: Providing a description for groups of form controls using fieldset and legend elements. The fieldset element contains a legend element (that should be the first child of the fieldset) which provides an identification for the field group.

The W3C acknowledges that the default presentation of the fieldset element in most browsers is somewhat tacky, but also advise against completely removing the visual grouping provided by a fieldset, stating:

Authors sometimes avoid using the fieldset element because of the default display in the browser, which draws a border around the grouped controls. This visual grouping is also useful and authors should seriously consider retaining it (or some form of visual grouping). The visual effect can be modified in CSS by overriding the "border" property of the fieldset and the "position" property of the legend.

Techniques for WCAG 2.0: H71: Providing a description for groups of form controls using fieldset and legend elements

Fields will usually be grouped because they form part of a larger logical unit. For instance, the fields that make up a billing address should be grouped by a fieldset. However, this also extends to what may initially be thought of as single data items that are nonetheless broken apart into multiple fields, such as:

  • A date data item being entered by way of separate fields for day, month and year; or
  • A phone number data item, split into an area code and local number.

Grouping these fields would allow a user with supporting assistive technology to understand the purpose of the entire data item, and skip it if necessary. The latter example could be exposed to such users as a "home phone number", for instance, letting them skip it if necessary. Each input control should also be labelled according to its individual purpose, possibly using the title attribute if the label is not to appear to sighted users by default.

Indeed, one common failure listed for Success Criterion 3.3.2 (Labels or Instructions) specifically relates to formatting a phone number as multiple fields, but without including a text label. The failure uses the North American structure of a phone number as three fields, but the concept is there regardless.

Recent changes in this test mean that fieldsets are optional "when a group of related radio buttons or checkboxes (even having values for a single named field) includes clear instructions and distinct selections." This means if each selection is explained fully in a label, a fieldset is not required. Starting from Version 49 of the HTML_CodeSniffer bookmarklet, errors related to the omission of a fieldset have been downgraded from Error to Warning.

For instance, if you have a "delivery option" field that has two options simply labelled "Delivered" and "Pick Up", you would probably need a fieldset - the option labels would not make sense out of context, and need the context of the fieldset legend to fully make sense. If the fields were labelled, say, "I would like this item delivered to me" and "I wish to pick this item up" - offering clear and distinct explanations of each option - a fieldset would likely not be required. This is a judgement call that has to be made by the manual tester, based on whether the labels would make sense by themselves.

Grouping drop-down options using the optgroup element

Technique H85: Using OPTGROUP to group OPTION elements inside a SELECT suggests that groups of related option elements inside a select element (drop-down field) should be grouped using the optgroup field.

Marking up lists

Where a set of items is written as a list, they should be marked up using the appropriate list type, as per technique H48: Using ol, ul and dl for lists or groups of links. There are three list types:

  • unordered (bulleted, like this one), using the ul element;
  • ordered (numbered/lettered), using the ol element; and
  • definition (where a term is followed by its definition, usually indented).

Most checks (especially for a definition list) will likely need visual confirmation that the block should be a list, however HTML_CodeSniffer will throw errors where certain paragraphs appear to be a list but are not marked up as such.

Paragraphs broken up by line breaks (br elements) will be detected as being potential lists when at least one line starts with the following:

  • an asterisk or dash, indicating an unordered list, or
  • a number followed by spacing, possibly including a punctuation mark between the two (eg. "1." or "1/" or similar), indicating an ordered list.

HTML_CodeSniffer will suggest that these elements should be converted into the appropriate list type.

Avoiding presentational markup

HTML_CodeSniffer will throw errors where authors use so-called "presentational markup" as defined in HTML 4.01. Specifically, it tests for these presentational elements and attributes:

Note that use of elements producing an underline are generally discouraged, or a change in their display recommended, because most users expect hyperlinks to be underlined.

In HTML5, most such markup has been given a semantic meaning, usually defining what type of text normally uses the default formatting, without conveying an additional sense of emphasis as the strong or em elements do.

Nesting of headings

HTML_CodeSniffer will emit messages when headings are not nested in the correct order. For example, the primary heading of the document not being a H1, or heading levels being skipped such as a H1 being followed by a H3.

At A and AA level, they will emit a warning, as there are examples in the HTML version of the technique involved (H42: Using h1-h6 to identify headings) which suggest that headings don't necessarily have to be in order.

At AAA level, an error will be thrown instead, with the reason related to Success Criterion 2.4.10 - which states that "section headings are used to organise the content". With incorrectly-nested headings, it makes it more difficult for some users to grasp the overall structure of the page, and so it is considered important that they are properly used. The single relevant technique for that Success Criterion states:

To facilitate navigation and understanding of overall document structure, authors should use headings that are properly nested (e.g., h1 followed by h2, h2 followed by h2 or h3, h3 followed by h3 or h4, etc.).

Techniques for WCAG 2.0: G141: Organizing a page using headings

Therefore, this is enforced at the AAA level.
