Success Criterion 3.2.2: On Input (Level A)

Changing the setting of any user interface component does not automatically cause a change of context unless the user has been advised of the behavior before using the component.

(From Understanding WCAG 2.0: Understanding Success Criterion 3.2.2)

Discussion

Summary

The main purpose of this Success Criterion is ensuring that users do not become confused by an unexplained change of context arising from an input change in a form.

Using a submit button to change context in a form is the best possible method. If this is not possible and a change in input must result in a change of context, then there should be instructions stating explicitly what would happen once the input occurs - technique G13: Describing what will happen before a change to a form control that causes a change of context to occur is made.

This instruction material must be placed in a location readily accessible to all users. Placing the instruction material in a bypassable location is considered a common failure, documented as failure F76.

Tests for submit buttons

A submit button is the most appropriate control to use to signal to a user that the button will explicitly trigger a change in context. This is preferable to the context change being placed out of their control by a web page. (That this criterion is successful depends, of course, on the author not doing such latter things.)

This is covered by a general technique (G80: Providing a submit button to initiate a change of context) and a HTML-specific technique (H32: Providing submit buttons).

In HTML, the three elements that count as submit buttons are:

  • input element with type "submit" - providing a normal button that submits.
  • input element with type "image" - providing an image submit button.
  • button element with type "submit" - providing a normal submit button that can be more richly styled than the equivalent input element.
