Success Criterion 3.2.5: Change on Request (Level AAA)

Changes of context are initiated only by user request or a mechanism is available to turn off such changes.

(From Understanding WCAG 2.0: Understanding Success Criterion 3.2.5)

About opening links in a new window

Pop-up windows are included in this Success Criterion. When links are opened in a new window, users should know that this is going to happen. The most common method is using the target attribute to open a new window on user request and indicating this in link text - setting it to "_blank" to indicate that it opens in a new window.

Use of the target attribute

The target attribute is not supported by all W3C web document types - notably, it was removed from HTML 4.0 Strict (and thus, XHTML 1.0 Strict). This is because its original usage was as a target for frames in a frameset, and framesets were not included in the HTML 4.0 Strict standard.

However, the target attribute has been re-introduced into HTML5. Although framesets are considered obsolete and non-conforming, the more pervasive support for inline frames mean that the target attribute maintains relevance in current web development. (Target names are referred to as "browsing contexts" in HTML5.) Target attributes also have the advantage of being machine-readable:

Use of the target attribute provides an unambiguously machine-readable indication that a new window will open. User agents can inform the user, and can also be configured not to open the new window.

Techniques for WCAG 2.0: H83: Using the target attribute to open a new window on user request and indicating this in link text

In recent browsers, a "new window" may also include a new tab.

Where a link is specified using target="_blank", HTML_CodeSniffer will test for whether the link mentions that it will open in a new window. If the words "new window" (not case-sensitive) aren't found in the link name, a warning will be generated.

Since this is a Success Criterion applying at Level AAA, the fact that something opens in a new window should be part of the link title, and not part of the context - ie. the surrounding sentence or list item. This is because Success Criterion 2.4.9 would also apply, which requires the link text to describe the purpose without assistance from its context.

Using progressive enhancement as an alternative

Where target is not supported, you can use technique SCR24: Using progressive enhancement to open new windows on user request so user agents with JavaScript can modify the links to open in a new window, while other user agents would gracefully fall back to the original HTML code, which opens the link in the same window. HTML_CodeSniffer cannot detect links which open a new window in this way.
