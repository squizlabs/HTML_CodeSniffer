Success Criterion 4.1.1: Parsing (Level A)

In content implemented using markup languages, elements have complete start and end tags, elements are nested according to their specifications, elements do not contain duplicate attributes, and any IDs are unique, except where the specifications allow these features.

Note: Start and end tags that are missing a critical character in their formation, such as a closing angle bracket or a mismatched attribute value quotation mark are not complete.

(From Understanding WCAG 2.0: Understanding Success Criterion 4.1.1)

Summary

The concept being described in this Success Criterion is similar to that of a document being "well-formed", as it is defined in XML. If a document is not well-formed, a user agent will either completely reject the document, or attempt to repair it - something that may not be interpreted the same way across browsers.

Note the rider "except where the specifications allow these features". This is to allow pages that conform to the HTML specification to pass this Success Criterion - as it does not require closing tags for many elements (or in some cases, even opening tags - for instance, tbody). This is why the three HTML-specific techniques exist:

XHTML documents

For XHTML documents, technique H75: Ensuring that Web pages are well-formed applies. Technically speaking, documents that are not well-formed will not render at all in browsers, but that does not tend to apply when the document is sent with a HTML content type.

HTML documents

Because HTML serialisation does not have a concept of being "well-formed" so to speak, following technique H88: Using HTML according to spec is sufficient. Otherwise, at an absolute minimum, the following three techniques must be followed:

  • H74: Ensuring that opening and closing tags are used according to specification
  • H93: Ensuring that id attributes are unique on a Web page
  • H94: Ensuring that elements do not contain duplicate attributes

What can HTML_CodeSniffer check for?

HTML_CodeSniffer makes checks for duplicate ID attributes. These errors are flagged as a Common Failure F77: Failure of Success Criterion 4.1.1 due to duplicate values of type ID. The failure is defined this way because the failure is generic, and attributes can be defined as "type ID" for other types of XML documents through a document type definition (DTD) or XML Schema.

Note: A lack of failures of this kind can only be considered authoritative if a complete document is processed. A document fragment can be tested for duplicate IDs, but it does not guarantee conformance if a fragment is inserted into a document, as it may duplicate IDs used by the fragment.

Other errors in this category cannot currently be tested - and would be impossible to test whenever HTML_CodeSniffer is passed a document object model (DOM) element rather than source code. This is because any source code provided from the user agent will generally be reconstructed from the DOM, and therefore will not be the same as the original source code.
