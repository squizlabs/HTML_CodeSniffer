Success Criterion 4.1.2: Name, Role, Value (Level A)

For all user interface components (including but not limited to: form elements, links and components generated by scripts), the name and role can be programmatically determined; states, properties, and values that can be set by the user can be programmatically set; and notification of changes to these items is available to user agents, including assistive technologies.

Note: This success criterion is primarily for Web authors who develop or script their own user interface components. For example, standard HTML controls already meet this success criterion when used according to specification.

(From Understanding WCAG 2.0: Understanding Success Criterion 4.1.2)

About Accessibility APIs

This goal often speaks of exposing certain data to an "accessibility API". Nominally speaking, this translates UI components to a set of data (such as role, name, value, state, available actions, etc) that can be communicated to a user via assistive technology such as a screen reader.

For HTML, the browser usually communicates with an OS-level accessibility API, which then exposes information to assistive technology. For a list of accessibility APIs in common use, the HTML to Platform Accessibility APIs Implementation Guide - which covers HTML 4 and 5 - is a good start. Other technologies such as Java and (presumably) Flash have techniques that expose their interfaces to accessibility APIs.

Using technique H91: Using HTML form controls and links successfully ensures that ordinary HTML form controls and links are accessible.

Complex components and WAI-ARIA

If you are creating custom controls and widgets, it may not be so easy to expose such controls to an accessibility API. For instance, a two-state on/off toggle button (like that found in the iOS interface) has the role of a check box - as it is representing an yes/no decision - and thus should be exposed to assistive technology as such, rather than being dependent on how it is implemented.

The WAI-ARIA document attempts to address this with a common vocabulary of roles, which would then be implemented using the role attribute of a host language, such as XHTML or HTML5.

XHTML can be extended through the use of namespaces, however HTML 4.01 cannot, and thus the use of the role and WAI-ARIA-specific attributes result in validation errors against the usual DTDs. It is suggested that "an enhanced schema or DTD be created for each host language" and this may include HTML 4.01.

The WAI-ARIA document is currently only at Candidate Recommendation stage, pending further implementation and testing. As of the time of writing, there has not been much progress towards completion of this stage.

Anchor elements without href attributes

An anchor ("a") element can be used without a href attribute, thereby defining a destination for a link (a "destination anchor"). However, in HTML_CodeSniffer, this will emit a warning, because in certain versions of HTML this conflicts with the element's intended role.

HTML 4.01 and XHTML 1.0

In HTML 4.01, anchors are described as either being a "source" or "destination" anchor. A href attribute turns an anchor into a "source" anchor - creating the links we all know and love. Adding a name attribute to an anchor - or an id attribute to any element - creates a "destination" anchor, which can then be targeted through a source anchor.

Source anchors have the role of a link, and thus, must have link text. However, destination anchors are considered "merely" anchors, and do not have such a role. Therefore, the requirement that they must have link text does not apply, as they do not have to expose a name to the accessibility API.

For forward compatibility, HTML_CodeSniffer recommends that you consider placing the ID or name attribute of the anchor on a nearby element instead, and remove the destination anchor. For instance, instead of placing an anchor at the start of the contents of a heading element, place the anchor's ID on the heading instead.

The same applies in XHTML 1.0, as it is specifically a reformulation of HTML 4.01 in XML, hence the same definitions apply.

HTML5

In HTML5, an "a" element without a href attribute (but with content) is considered legal:

If the a element has no href attribute, then the element represents a placeholder for where a link might otherwise have been placed, if it had been relevant.

(W3C: HTML5, "The a element")

An example is given of a navigation section where the pages in a section are placed in "a" elements, but the current page's anchor does not contain a href, as a page pointing to itself in navigation is not considered good practice.

However, the above definition also means that using "a" elements as destination anchors a la HTML 4.01 is not correct - it means something different. Similarly, the use of the name attribute is considered "obsolete but conforming", meaning it will have no effect. For correctness, "a" elements should only be used for links or potential links, and the ID attribute should always be used (rather than name). For other targets, place the ID attribute on the intended target itself.
