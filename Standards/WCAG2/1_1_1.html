Success Criterion 1.1.1: Non-text Content (Level A)

All non-text content that is presented to the user has a text alternative that serves the equivalent purpose, except for the situations listed below.

  • Controls, Input: If non-text content is a control or accepts user input, then it has a name that describes its purpose. (Refer to Guideline 4.1 for additional requirements for controls and content that accepts user input.)
  • Time-Based Media: If non-text content is time-based media, then text alternatives at least provide descriptive identification of the non-text content. (Refer to Guideline 1.2 for additional requirements for media.)
  • Test: If non-text content is a test or exercise that would be invalid if presented in text, then text alternatives at least provide descriptive identification of the non-text content.
  • Sensory: If non-text content is primarily intended to create a specific sensory experience, then text alternatives at least provide descriptive identification of the non-text content.
  • CAPTCHA: If the purpose of non-text content is to confirm that content is being accessed by a person rather than a computer, then text alternatives that identify and describe the purpose of the non-text content are provided, and alternative forms of CAPTCHA using output modes for different types of sensory perception are provided to accommodate different disabilities.
  • Decoration, Formatting, Invisible: If non-text content is pure decoration, is used only for visual formatting, or is not presented to users, then it is implemented in a way that it can be ignored by assistive technology.

(From Understanding WCAG 2.0: Understanding Success Criterion 1.1.1)

Alternatives for non-text content

A large proportion of the tests performed for Success Criterion 1.1.1 relate to text alternatives for images, objects such as Flash movies, and other "non-text content" - the latter fully defined as:

any content that is not a sequence of characters that can be programmatically determined or where the sequence is not expressing something in human language Note: This includes ASCII Art (which is a pattern of characters), emoticons, leetspeak (which uses character substitution), and images representing text

Understanding WCAG 2.0: Understanding Success Criterion 1.1.1 - definitions section

In one way or another, non-text content should be provided with a text alternative that can effectively substitute for the non-text content when not available - for instance, when using a screen reader which cannot "see" the true content of images, or where a required plugin may not be installed (or even available) for certain complex objects. For instance, replacing Flash content with simply "you need to install the Flash plugin" is insufficient - it means users unable to use Flash have no ability to access your content.

Note: If the non-text content is simply repeating what is primarily available in an accessibility-supported way, then its content is already described, in a roundabout way: the body text of the web page being a long text alternative.

  • Example: a web page which contains the details of a marketing brochure is supplemented with a copy of that brochure in a PDF document. If the contents of the two are the same, then this should be considered sufficient.

This is similar to the "media alternative for text" exemptions found in the time-based media criteria of WCAG 2.0, such as Success Criterion 1.2.1.

Short and long text alternatives

The WCAG specification makes mention of two "lengths" of text alternatives: short and long. This passage generally describes where a long-text alternative should be used:

If possible, the short text alternative should completely convey the purpose and information. If it is not possible to do this in a short phrase or sentence, then the short text alternative should provide a brief overview of the information. A long text alternative would be used in addition to convey the full information.

Techniques for WCAG 2.0: G94: Providing short text alternative for non-text content that serves the same purpose and presents the same information as the non-text content

An example would be providing a chart of, say, downloads figures for HTML_CodeSniffer over the course of a year. The short text alternative would say, for instance, "Graph of HTML_CodeSniffer downloads by month over the past year". Following this would be, for instance, a table of the actual download numbers, or a link to a page containing such a table. Alternatively, the long description would provide a summary of the trends that the graph was included to show.

The methods of providing short and long text alternatives are summarised in Understanding Success Criterion 1.1.1.

The longdesc attribute should be avoided for providing a long text alternative. Although screen readers are usually able to access it, browsers in general do not support it on their own, thereby depriving other classes of users - for instance, those with mild visual disabilities and use zooming instead of a screen reader. Better alternatives are to include it in the body text after the non-text content, or to provide a link to a page with a long description.

Decorative images

All images must contain a short text alternative to satisfy Success Criterion 1.1.1, however it is possible for the text alternative to be empty. If the image does not convey information, the text alternative makes sense to be blank.

HTML_CodeSniffer will emit a warning for each image that contains an empty alt attribute, so authors ensure that the images are indeed decorative. The title attribute must also be either absent or empty, otherwise. An attribute consisting entirely of whitespace is also considered "empty".

Another alternative to including purely decorative images is through use of CSS (see Technique C9: Using CSS to include decorative images).

Technique C9 reminds authors and designers that including images through CSS is "not appropriate for any image that conveys information or provides functionality, or for any image primarily intended to create a specific sensory experience". Style sheets can always be overridden with a user-specific style sheet, or ignored completely or partially in certain situations - eg. in a screen reader scenario, where the "speech" media type would ignore image-related CSS rules.

Appropriate text alternatives

This section describes appropriate short and long text alternatives for each possible type of non-text content.

Standalone img elements

The purpose of a standalone img element should be summarised in a short text alternative, presuming the image is not purely decorative.

Note that it's the purpose that should be described, not necessarily a description of the image:

[Example:] A picture shows how a knot is tied including arrows showing how the ropes go to make the knot. The text alternative describes how to tie the knot, not what the picture looks like.

Techniques for WCAG 2.0: G94: Providing short text alternative for non-text content that serves the same purpose and presents the same information as the non-text content

Linked img elements

When the img element is linked to another page or fragment, the rules of Success Criterion 2.4.4 and Success Criterion 2.4.9 override the previous section: the contents of the link must describe the purpose of the link. Where the img element is the only contents of the link, this means its short text alternative (ie. alt text) must describe the link's purpose:

[Example:] A search button uses an image of a magnifying glass. The text alternative is "search" and not "magnifying glass".

Techniques for WCAG 2.0: G94: Providing short text alternative for non-text content that serves the same purpose and presents the same information as the non-text content

Where an img element is combined with text, the combined contents of the two would describe the link's purpose. It is possible in this context that the image adds no further information to the text, and thus can be marked "decorative" with an empty alt text. Certainly, the text and image should not say the same thing - see "Link stuttering" below.

Image submit buttons

Image submit buttons follow the same rules as images linked in "a" elements: their text alternative should describe the purpose of the button, not necessarily describe the image.

Client-side image maps

Appropriate accessibility in client-side image maps (using the "map" element) are covered in technique H24: Providing text alternatives for the area elements of image maps.

Each "area" element of a client-side image map should contain alt text as to the purpose of the link created by the image (as these are links). Alternatively, the web page can use a redundant mechanism (such as a list of links) to provide descriptions of the areas in the map.

The latter method may be preferred due to inconsistent user support for alt attributes in image maps - particularly in cases of keyboad navigation, or when images are disabled.

Object elements

Embedded objects (using the "object" element), such as Flash movies, Java applets, and so on, allow both text and non-text alternatives to be provided as fallback content. For instance, a movie may be replaced with a static image, which may itself provide alt text (if provided using img) or its own fallback content (if using nested object tags).

In these cases, it is important that an alternative be provided to the contents of the object element. This may be using the body of the innermost object element, or by other methods of providing a long text alternative. HTML_CodeSniffer produces an error when a non-text alternative cannot be found in an object element group, as this is most likely to be a mistake. However, other long text alternative methods should also be sufficient.

Applet elements

Although non-standard (except for very early versions of HTML), applet elements are still reasonably well supported throughout most browsers. The W3C standard recommends that the object element be used instead.

Nonetheless, the Techniques for WCAG 2.0 document contains one technique regarding the use of the applet element. This element is slightly more complicated when it comes to text alternatives than the object element, since it has two separate alternatives:

  • A body text section, for providing an alternative where the rendering interface of the browser (eg. Gecko, Webkit) does not recognise the applet tag without a plugin (and the plugin is not there), or
  • An alt attribute, for browsers whose rendering interface recognises the applet tag built-in (Internet Explorer in particular) but cannot load the applet.

Because the decision as to which alternative is used is inconsistent between browsers, these locations should use the same, short text alternative. A long text alternative can be provided through the usual methods such as body text or a link.

Link combining and stuttering tests

When an image and adjacent text are to point to the same location, they should be combined into one link. Sometimes, authors split them into two separate links to avoid having, say, an underlined character between the two. However:

Visually they appear to be the same link, but they are experienced by many people as two identical links and this can be confusing.

Techniques for WCAG 2.0: H2: Combining adjacent image and text links for the same resource

Two incorrect applications of this technique include:

  • Keeping the two links, but removing the alt text from the image. This results in a link with no text at all, leading to undefined behaviour.
  • Combining the links, but keeping the alt text identical with the text content. This results in a link where the link text appears to be stated twice.

Note: The alt text does not always have to be completely removed. For example, a link may contain a combination of a file name and a type icon indicating it as a MS Word Doc file, then the icon can say that it is of a MS Word Doc type. If it says this in the link text however, then this would not be appropriate.

Explicitly not tested

We do not test Technique H46: Using noembed with embed. This technique speaks of marrying the non-standard embed element, used by as a forerunner to the HTML 4.0-standard object element, with a noembed element which provides a text alternative. The noembed element could be placed either inside the embed element, or immediately afterwards.

It is arguable that the noembed tag cannot be used in an accessible document, for the following reasons:

  1. Prior to HTML5, the embed element itself was non-standard. Although this alone does not fail Success Criterion 4.1.1 (Parsing) - validation is not a requirement, only well-formedness is - it makes it extremely difficult to satisfy the sufficient techniques. This Success Criterion suggests ensuring opening and closing tags are used according to specification as a technique for HTML serialisation (as well-formedness is not a proper concept in HTML), but embed is not in the HTML spec, does this mean it is impossible to use the tag in HTML 4.0 and pass this Success Criterion?
  2. In HTML5, the embed element is included, however the noembed element is absent. The spec states that the embed element does not include fallback content; user agents should use a default plugin instead (even if to say "this content is unsupported"). Therefore, there is no guarantee that the default content will be accessible.

Accessible documents should use the object element instead.
