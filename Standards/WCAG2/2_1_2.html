Success Criterion 2.1.2: No Keyboard Trap (Level A)

If keyboard focus can be moved to a component of the page using a keyboard interface, then focus can be moved away from that component using only a keyboard interface, and, if it requires more than unmodified arrow or tab keys or other standard exit methods, the user is advised of the method for moving focus away.

Note: Since any content that does not meet this success criterion can interfere with a user's ability to use the whole page, all content on the Web page (whether or not it is used to meet other success criteria) must meet this success criterion. See Conformance Requirement 5: Non-Interference.

(From Understanding WCAG 2.0: Understanding Success Criterion 2.1.2)

Discussion

Summary

The purpose of this Success Criterion is primarily one of non-interference. If a user cannot navigate using a mouse, plugins can pose a problem to navigate. This Success Criterion ensures that all plugins - whether relied on for compliance or not - have a keyboard-related method of exiting and returning to the text content surrounding it.

Generally speaking, a plugin must have ways of exiting to the content both before and after the position at which it is embedded. The conventional ways of doing this would either be with the arrow keys, or with the Tab key (and Shift-Tab). Any non-standard combination must be exposed to the user. Providing one method of exiting, which only allows exiting in one direction, would likely be construed as interference, as this does not provide the user with free access to the content on either side of the plugin.
