Success Criterion 2.4.1: Bypass Blocks (Level A)

A mechanism is available to bypass blocks of content that are repeated on multiple Web pages.

(From Understanding WCAG 2.0: Understanding Success Criterion 2.4.1)

Discussion

About varying message levels on fragment link tests

Success Criterion 2.4.1 contains a test that emits a message when an anchor (A) element contains a link to a fragment in the same document (eg. "#top", "#content"). These links are often used to skip to content, and as such can be very useful for them to be correct. For instance, users with screen readers, or sighted users requiring keyboard access, depend on skip links to reach a page's content quickly, so their inclusion is good accessibility practise, and if incorrect can result in user disorientation or frustration as they are either sent to an unexpected part of the page (such as the top), or nowhere at all.

Following the inclusion of Pull Requests #10 and #11, documents that contain at least a body element will still emit an error if the ID of a fragment link cannot be found. Other document fragments will emit a warning only, if that ID cannot be found in the tested fragment.

Although both messages should be acted upon, document fragments will only emit a warning because it is non-authoritative: whether the code would conform in a document depends on the other contents of a document. For instance, the source of a paragraph or other content section may be submitted to HTML_CodeSniffer for evaluation. The paragraph contains a "back to the top" fragment link (to "#header", for instance), and this could point to an element named "header" at the top of the page.

If the paragraph is tested in isolation, a message will appear, but if included within the page with that ID, the document would pass. Conversely, if the same paragraph was placed into a different design without a "header" ID, it would rightly fail (and if tested as a full document, would emit an error). Therefore, the absence of the ID cannot determine a failure of conformance to this success criterion, so is emitted as a warning to ensure authors properly check the link in context.

The reasoning behind this is similar to why conformance can only be claimed for full web pages, as per Conformance Requirement 2 of the WCAG 2.0 recommendation: a fragment can only be considered in the context of a full page for conformance purposes.

Changelog for the fragment links test:

  • Added on 17 May 2012 (Pull Request #5).
  • Document fragments now only emit a warning as of 7 June 2012 (Pull Requests #10, #11).
