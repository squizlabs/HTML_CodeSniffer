Success Criterion 3.2.1: On Focus (Level A)

When any component receives focus, it does not initiate a change of context.

(From Understanding WCAG 2.0: Understanding Success Criterion 3.2.1)

Discussion

Summary

The purpose of this Success Criterion is to ensure that no unexpected changes of context occur as a mere result of a form control receiving the focus. Users should be able to navigate through the page, by mouse or keyboard, without seeing things done to their browsing context which include:

  • their focus being switched to another component,
  • forms automatically submitted, or
  • new windows automatically opened.
Examples

  • Automatic pop-up windows, such as those used for advertising, fail the success criterion as per failure F52. "Pop-under" ads technically do not fail this due to the window not automatically being given focus, however if the ad is forced to the front programmatically on close, it could possibly argued that this action still fails.

  • For elements such as links or buttons, it is tempting to set an event handler on the "focus" event, so it works with both mouse and keyboard. However, users not using the mouse to navigate through the document will find that the event will fire as soon as the element is navigated to, without user intervention. Instead, the event should fire on element activation - in these cases, the "click" event will work as generic activation event. (See G107: Using "activate" rather than "focus" as a trigger for changes of context).
