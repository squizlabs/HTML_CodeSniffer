Success Criterion 2.1.1: Keyboard (Level A)

All functionality of the content is operable through a keyboard interface without requiring specific timings for individual keystrokes, except where the underlying function requires input that depends on the path of the user's movement and not just the endpoints.

Note 1: This exception relates to the underlying function, not the input technique. For example, if using handwriting to enter text, the input technique (handwriting) requires path-dependent input but the underlying function (text input) does not.
Note 2: This does not forbid and should not discourage providing mouse input or other input methods in addition to keyboard operation.

(From Understanding WCAG 2.0: Understanding Success Criterion 2.1.1)

Note

Technically superseded by Success Criterion 2.1.3 at level AAA. To avoid redundancy, keyboard-related warnings will appear to come from Success Criterion 2.1.1 regardless of level.

Redundant event handlers

The main focus of this Success Criterion is to employ either device-independent events that will work with both keyboard and mouse, or to provide redundant keyboard events. The relevant technique is SCR20: Using both keyboard and other device-specific functions. Events which will emit messages if found include mouseover, mouseout, mousemove, mousedown, mouseup, and dblclick.

If the process cannot be replicated exactly using keyboard events, some other way must be provided to perform the action (technique G90: Providing keyboard-triggered event handlers). For example, a mouse interface may use a drag-and-drop method for moving items between certain areas of a page. Although the "drag and drop" cannot be replicated by the keyboard, alternative actions can be provided: for instance, arrow keys to select, tab to switch between areas, and "cut" and "paste" shortcuts to move the items.

What about the click event?

You will notice that the above list does not include click events. This is because the click event is also fired as part of an activation trigger (see technique SCR35: Making actions keyboard accessible by using the onclick event of anchors and buttons).

For instance, when browsing a page using the keyboard, a link produced from an "a" element can be activated by using the Enter key. This will generate the relevant key events for that keypress, but will also generate a click event (referred to as a "synthesised" click event. Similarly, if an accessibility API triggers its activation, a click event will be generated regardless of how it implements the activation.

The DOM Level 3 events specification draft states:

Though the click event type has its origins in pointer devices (e.g., a mouse), subsequent implementation enhancements have extended it beyond that association, and it can be considered a device-independent event type for element activation. (Click event type)

If the instance of the activation trigger is not an event of event type click[, ...] the implementation must synthesize and dispatch an event of event type click as one of the default actions of that activation trigger; the value of the Event.target must be set to the event target (normally, the currently focused element), and the event must simulate a left click... (Activation event synthesis)

Document Object Model (DOM) Level 3 Events Specification (working draft, hence work in progress - 14 June 2012 version quoted)

This definition appears to deprecate the use of the DOMActivate event, introduced in the DOM Level 2 Events specification, which was never strongly supported by browsers (and not by Internet Explorer).

Which event implementations are tested?

Events can be implemented using a variety of different methods - ranging from the HTML attributes and JavaScript properties, through to DOM events (eg. addEventListener), or being set through frameworks like jQuery or Prototype.js. On the flipside, even if a corresponding event is provided in code as an HTML attribute, there is no guarantee that it exposes the same functionality as the mouse event.

The main stumbling blocks to being able to test all events are:

  • The processing time that it would take to traverse every element in the document and determine if each element has any of the mouse-bound JavaScript events (set through element.onmousedown, element.ondblclick, etc), and
  • The lack in DOM Level 3 of an ability to determine whether an event listener has been added through addEventListener(). Many frameworks will wrap the event handler system, providing tracking of the events that have been attached in this way.

Currently, a warning is generated on elements that contain the mouse-specific attributes. It will be up to the author to ensure that the functionality is provided to keyboard users in an appropriate manner.
