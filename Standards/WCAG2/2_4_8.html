Success Criterion 2.4.8: Location (Level AAA)

Information about the user's location within a set of Web pages is available.

(From Understanding WCAG 2.0: Understanding Success Criterion 2.4.8)

Summary

This Success Criterion is best described with its one-sentence intent statement:

The intent of this Success Criterion is to provide a way for the user to orient herself within a set of Web pages, a Web site, or a Web application and find related information.

Understanding WCAG 2.0: Location: Understanding SC 2.4.8

About the link element

Using the link element to programmatically relate pages is covered in technique H59: Using the link element and navigation tools. Some user agents provide methods for using these links, but this has suffered a little due to lack of consensus on some standard link relation types.

The link element includes several different relation types or roles. Beyond stylesheet and "alternate" representations (for RSS/Atom feeds, for instance), some relations are used for navigation purposes.

HTML_CodeSniffer will test ALL link elements for validity, not just those specified for navigational techniques. That is, they will be checked to ensure they contain a valid rel attribute and value attribute.

Relation types are defined in multiple places:

  • The XHTML vocabulary which defines roles as they relate to XHTML+RDFa, for XHTML 1.1+.
  • The IANA link type registry as it applies to RFC 5988: Web Linking. This proposes that links would be able to be expressed as HTTP headers.
  • The HTML5 specification contains a base list of supported relation types - note that only "next" and "prev" are defined for navigation purposes; other proposed link types in HTML5 are explicitly the responsibility of a HTML5 link type registry maintained by the Microformats community.
Support for navigation links in browser chrome

Although older browsers may have displayed navigational link elements in their user interface "chrome", it is no longer a popular option, and has been removed from later browsers. Some support add-ons which add this support back in. In summary:

  • Opera (as of version 12.00) includes a "Fast Forward" feature, accessed by pressing the spacebar at the end of a page containing a "next" relation. A "Next" button is also available for the toolbar. A Navigation Bar used to provide the other relations, but this was removed in version 12.00.
  • The "Suite" versions of Mozilla/Netscape used to handle it, and SeaMonkey retains this feature (View menu - Show/Hide - Website Navigation Bar).
  • Firefox (and probably IE) have extensions that expose link relations in a separate toolbar.

The "next" relation is also used for link prefetching in Firefox (and probably later versions of SeaMonkey), as well as the explicitly specified "prefetch" relation.

Direction of links

HTML5 only supports "forward" links (using the rel attribute). The concept of "reverse" links (using the rev attribute), as found in earlier versions of HTML, are not supported. HTML_CodeSniffer will, similarly, not recognise link elements with reverse relations.

The only major reverse relation that was commonly used in HTML was rev="made", which referred to the author of the page. This has been changed to rel="author" in HTML5 so it can be expressed as a forward relation.

Additional sufficient techniques

Beyond the link element, the following techniques are also sufficient for allowing a user to determine the location of a page, where appropriate:

  • Providing a breadcrumb trail.
  • Providing a site map.
  • Indicating current location within navigation bars - but, just changing the background colour is not sufficient (see Success Criterion 1.4.1 (Use of Colour)): something else should change, such as disabling the link, bolding the text, or adding an indicator such as an asterisk or bullet.
