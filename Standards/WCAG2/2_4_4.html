Success Criterion 2.4.4: Link Purpose (In Context) (Level A)

The purpose of each link can be determined from the link text alone or from the link text together with its programmatically determined link context, except where the purpose of the link would be ambiguous to users in general.

(From Understanding WCAG 2.0: Understanding Success Criterion 2.4.4)

Note

Superseded by Success Criterion 2.4.9 at AAA level, which removes the link context.

Summary

The purpose of this Success Criterion is to ensure that link text, combined with their immediate surrounding context, provide enough information for letting users decide whether to follow the link.

This is particularly important when many assistive technologies offer the ability to provide a list of links without browsing through the whole page. Preferably, the link text itself would be enough to ensure this decision is possible (and this is a requirement of Level AAA through Success Criterion 2.4.9), but at Level A it is permitted to take into account the immediate surrounding context.

Why the "ambiguous" exception?

This Success Criterion applies to all links "except where the purpose of the link would be ambiguous to users in general". This is defined as when:

the purpose cannot be determined from the link and all information of the Web page presented to the user simultaneously with the link (i.e., readers without disabilities would not know what a link would do until they activated it).

Example: The word guava in the following sentence "One of the notable exports is guava" is a link. The link could lead to a definition of guava, a chart listing the quantity of guava exported or a photograph of people harvesting guava. Until the link is activated, all readers are unsure and the person with a disability is not at any disadvantage.

WCAG 2.0 Recommendation: Definition - "ambiguous to users in general"

In other words, if a well-functioning user would be just as confused by the link as a disadvantaged user, this criterion would not be violated, since the purpose of accessibility rules such as this is that users with disabilities are not at any disadvantage to normal users. That is not to say, however, that applying the techniques would not be a good idea - in this case for all users.

What is acceptable as surrounding context?

Link context is defined through a number of techniques for programmatically defining link context, which are provided in the message provided by HTML_CodeSniffer. These include combining the link text with:

  • the text in the enclosing list item (H77), or
  • the text in an enclosing paragraph (H78), or
  • the text in an enclosing table cell and, if applicable, associated table headings (H79), or
  • the preceding heading element (H80), or
  • the parent list item, if the link is part of a nested list (H81).
  • the anchor's title attribute, if present (H33).

A common failure (F63) usually occurs as a result of incorrectly understanding what counts as link context. For instance, in Failure Example 1, a lead paragraph of a news article is followed by a "Read More" link. However, the two are printed in separate paragraphs, and therefore the link loses the context provided by the lead. If the "Read More" link was at the end of the lead rather than in a separate paragraph, the lead would provide suitable context to the "Read More" link. Alternatively, the link text could be changed to provide more context on its own.

Although provided (Failure Example 3) is an example of a definition list (dl element), which is provided because it could normally also be considered a candidate for providing a link context, but at the time of the recommendation's writing support for this was not strong, so could not be considered sufficient for satisfying this Success Criterion.
