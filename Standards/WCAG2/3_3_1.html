Success Criterion 3.3.1: Error Identification (Level A)

If an input error is automatically detected, the item that is in error is identified and the error is described to the user in text.

(From Understanding WCAG 2.0: Understanding Success Criterion 3.3.1)

Discussion

  1. Summary
  2. Use of presentation to display errors
  3. Use of browser-enforced validation

Summary

This Success Criterion ensures that all users are able to easily determine that a form submission has failed, by providing a text description of what field(s) are in error and why. Error messages should be as specific as possible; providing specifics of errors, whether they be missing required fields, or fields not in the correct format.

The relevant techniques - such as G83: Providing text descriptions to identify required fields that were not completed for missing required fields - suggest making the error as obvious as possible. The best practices offered in the techniques appear to be:

  • Provide a message above the form stating the errors that were found. This provides users who must read sequentially with early identification that the form is functioning but submitted with errors;
  • Adding some sort of general error identification to the title of the document, for the same reason;
  • Providing some sort of notice (preferably in a field's label) to denote that an error was found in that field, meaning users do not have to refer back to the initial message to find the other missing fields.

Use of presentation to display errors

This Success Criterion does not forbid the use of additional text styling or colour to further draw attention to the error. For instance, there is nothing wrong with highlighting fields in error in red for instance, as long as the actual error is described in text.

Use of browser-enforced validation

Some source languages allow validation to be carried out by the user agent, based on the base language itself. This is particularly pointed towards HTML5, where new input element types allow the browser to enforce client-side restrictions on inputs without requiring JavaScript. For example, some new browsers support the "pattern" attribute on text inputs, which can set a required format for text fields to follow; the pattern is the format of the PCRE subset supported by the JavaScript RegExp object.

This Success Criterion "does not require, nor prevent" the use of these new features. However, due to the patchy support for these in HTML5; their use alone to validate fields is not recommended.
